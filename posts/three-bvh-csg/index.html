Shapes that don't exist yet

Imagine designing a mechanical bracket — a block with two cylindrical holes drilled through it and a spherical pocket on top. Modelling by hand means placing hundreds of vertices. But a machinist describes it in one sentence: "box, subtract two cylinders, add a sphere."

That verbal description is CSG. Constructive Solid Geometry treats solids as sets of points and combines them with boolean operations. The result is always geometrically valid: no gaps, no orphaned triangles.

THE CORE QUESTION
Given two triangle meshes A and B, produce the mesh that represents A ∪ B, A ∩ B, or A − B.

CSG models are expressed as a binary tree. Leaves are primitives. Internal nodes are operations. The bracket above is just three nodes deep:

The bracket as a CSG tree. Leaves = primitives; nodes = operations.

What is CSG?

Every solid partitions 3D space into inside and outside. CSG combines two solids using set logic on their interiors:

A ∪ B = { p : p ∈ A OR p ∈ B } A ∩ B = { p : p ∈ A AND p ∈ B } A − B = { p : p ∈ A AND p ∉ B }

Beautiful in theory. Brutal in practice. You don't have a membership oracle — you have 50,000 triangles. The algorithm must determine which faces belong to the result, orient them correctly, and stitch everything into a valid manifold. That's what CSG implementations actually do.

The five operations

three-bvh-csg implements five core operations plus two "hollow" variants. Each one keeps a different subset of the source triangles.

∪ADDITION

Everything in A plus everything in B.

∩INTERSECTION

Only where A and B overlap.

−SUBTRACTION

A with B's footprint removed.

⇐REV. SUBTRACT

B with A's footprint removed.

⊕DIFFERENCE

XOR: non-overlapping parts only.

⊖HOLLOW SUB

Carves B's profile from A's surface only.

The old way: BSP trees

The classical CSG algorithm uses a Binary Space Partitioning (BSP) tree. You build a BSP for each brush (each input mesh), then merge the two trees. The result drops out of the merge traversal.

BSP-based CSG is elegant. It's also slow. Each polygon may need to be split by every partition plane it crosses. On a complex model this means quadratic growth in triangle count and time.

BSP CSG
  • Build BSP for each mesh — O(n log n)
  • Merge trees — O(n·m) in the worst case
  • Triangle count can explode during splits
  • Classic, well-understood, correct
BVH CSG (three-bvh-csg)
  • Build BVH for each mesh — O(n log n)
  • Find intersections via BVH–BVH query — prunes 99% of pairs
  • Only intersecting triangles need splitting
  • 100× faster on complex models
THE BIG IDEA
In most CSG operations the two meshes only partially overlap. The BVH finds the intersecting region instantly. Everything outside that region is classified by a single raycast per connected patch — no splitting needed.

Key insight: BVH-accelerated classification

To compute A − B you need to know, for every triangle in A, whether it is inside B or outside B. Inside → discard it. Outside → keep it (with correct orientation).

The naive test: pick a point on the triangle, shoot a ray in an arbitrary direction, count how many times it crosses B's surface. Odd = inside. Even = outside. This is a point-in-mesh test via raycasting.

Done naively that's O(n) per triangle — O(n²) total. With a BVH for B, each ray only tests O(log n) triangles. And better yet: non-intersecting triangles form connected patches. You only raycast one per patch, then flood-fill the classification through the half-edge graph to all neighbours.

One representative point per connected patch of A is raycast into B's BVH. The classification floods through the half-edge graph to all connected triangles.

Algorithm deep dive

The library's IMPLEMENTATION.md describes four phases:

1

Generate brush data

For each input mesh, build a BVH (spatial acceleration), a half-edge map (triangle connectivity), and a group index map (material assignment). These are cached — reuse them if the mesh hasn't changed.

2

Find all intersecting triangle pairs

Run a dual-tree traversal (bvhcast) between the two BVHs. This descends both trees simultaneously, pruning any node pair whose AABBs don't overlap. Only truly intersecting leaf pairs survive — typically a tiny fraction of all n×m pairs.

3

Classify non-intersecting triangles (flood fill)

For each connected patch of non-intersecting triangles, pick one representative, raycast it against the other mesh's BVH to get inside/outside, then flood through the half-edge graph to propagate the label to every triangle in the patch. O(triangles in patch) after the single raycast.

4

Split and classify intersecting triangles

For each intersecting pair, compute the intersection line segment and split both triangles along it. Each resulting sub-triangle is then classified individually via raycast. This is the bottleneck — each split triangle needs its own ray.

Half-edge flood fill

The half-edge map is the key to fast propagation. Every edge in a manifold mesh is shared by exactly two triangles. Storing both directions gives you instant triangle-to-triangle navigation.

The half-edge map stores twin relationships in typed arrays. Knowing T₀ is "outside" means T₁ — adjacent across the shared edge — inherits the same classification without another raycast.

Triangle splitting

When two triangles from different meshes intersect, the intersection is a line segment. Each triangle must be cut along that segment, producing sub-triangles that lie entirely inside or outside the other mesh.

The intersection segment splits T_A into sub-triangles. Each sub-triangle is then classified (inside/outside B) via raycast and kept or discarded per the selected operation.

The API in 5 minutes

The library is designed with a clear separation: Brush wraps geometry and caches its BVH, Evaluator performs the operation. 

import { SUBTRACTION, ADDITION, INTERSECTION, Brush, Evaluator } from 'three-bvh-csg';

// Create brushes — wraps THREE.Mesh, caches BVH internally
const brushA = new Brush( new THREE.SphereGeometry(1, 32, 32) );
brushA.updateMatrixWorld();

const brushB = new Brush( new THREE.BoxGeometry(1.2, 1.2, 1.2) );
brushB.position.set(0.5, 0.5, 0.5);
brushB.updateMatrixWorld(); // ← required before evaluate()

const evaluator = new Evaluator();
const result = evaluator.evaluate( brushA, brushB, SUBTRACTION );

// result is a regular THREE.Mesh — add to scene
scene.add(result);
Brush
  • Extends THREE.Mesh
  • Caches BVH + half-edge map
  • Cache is reused across repeated evaluations
  • Don't modify geometry after creation
Evaluator
  • Call .evaluate(a, b, op)
  • Returns a new THREE.Mesh
  • Or pass a target mesh to reuse buffers: .evaluate(a, b, op, target)
  • Thread-safe per instance

For hierarchical operations — a scene tree of brushes — use Operation and evaluateHierarchy:

import { Operation, OperationGroup, SUBTRACTION } from 'three-bvh-csg';

// Build a tree: box - sphere1 - sphere2
const root = new Operation( new THREE.BoxGeometry(2,2,2) );
const hole1 = new Operation( new THREE.SphereGeometry(0.5) );
const hole2 = new Operation( new THREE.SphereGeometry(0.5) );

hole1.operation = SUBTRACTION;
hole2.operation = SUBTRACTION;
hole1.position.set(-0.5, 0, 0);
hole2.position.set( 0.5, 0, 0);

root.add(hole1, hole2);
const result = evaluator.evaluateHierarchy(root);

The speedup grows with mesh complexity because BSP scales poorly: more triangles means more splits means more planes means more sub-triangles to process. BVH's logarithmic culling means classification cost grows slowly with n.

From 1977 to the browser

1977

Requicha & Voelcker

University of Rochester formalises CSG within the PADL solid modelling project. The CSG tree is defined as a mathematical object.

1980

BSP-based CSG

Fuchs, Kedem & Naylor introduce BSP trees for hidden-surface removal. The community quickly adapts them for CSG evaluation — elegant but O(n²) in the worst case.

1989

Ray-tracing CSG

POV-Ray ships with ray-tracing CSG: compute entry/exit intervals per primitive, intersect them. Fast for rendering, but outputs no mesh.

2002

Mesh-based CSG in games

Unreal Engine ships BSP-based brush tools. Level designers can subtract corridors from solid world geometry — the paradigm enters mainstream game development.

2016

Manifold (CGAL-quality)

Emmett Lalish starts Manifold — robust, numerically exact CSG using exact arithmetic and manifold guarantees. The CAD-grade baseline.

2022

three-bvh-csg

Garrett Johnson releases three-bvh-csg, powered by the BVH acceleration from three-mesh-bvh. Real-time, interactive CSG in the browser — 100× faster than BSP alternatives on complex geometry.

Limitations and open problems

MANIFOLD REQUIREMENT
Input geometry must be two-manifold (water-tight, no self-intersections, no degenerate triangles). Open meshes, planes, or badly-exported CAD files will produce wrong results.

Known issues from the IMPLEMENTATION.md:

  • Floating-point robustness — some triangles near coplanar regions aren't split correctly. Near-zero-area triangles can appear at intersection seams.
  • Split connectivity — each split triangle currently needs its own raycast. Once a triangle is split, the half-edge links are broken, so the flood-fill can't propagate through it. This is the main performance bottleneck on dense intersection seams.
  • No worker support (yet) — BVH and half-edge construction could run off the main thread; the framework is there but unused.
  • For CAD-grade robustness, prefer Manifold which uses exact arithmetic. three-bvh-csg trades precision for speed and interactivity.
